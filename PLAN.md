## フェーズ 1: 第2章「鉄道管理局の承認を得よう」

**学習内容**: `git remote add`, `git push`, プルリクエスト, レビュー, マージ承認, リリースタグ

### 1-1. ストーリーデータ（chapters.json に追加）

- 約10〜12シーン構成
- キャラクター: 建設局長、管理局審査官（新キャラ）
- シーン構成案:
  1. 導入: 自社線は完成したが、管理局の認可が必要
  2. `git remote add` — 管理局との直通運転協定を結ぶ（4択クイズ）
  3. ナラティブ: リモートの概念説明
  4. `git push` — ダイヤを管理局に提出（4択クイズ）
  5. ナラティブ: push後の状態
  6. プルリクエスト作成 — 運行申請を出す（4択クイズ）
  7. レビュー — 審査官がチェック（ナラティブ）
  8. レビュー指摘対応 — 修正してre-push（4択クイズ）
  9. マージ承認 — 申請が通る（ナラティブ）
  10. `git tag` — 開業日を記録（4択クイズ）
  11. 章まとめ
- **ポイント**: この章はスムーズに進む設計。issueの必要性はまだ出さない

### 1-2. グラフ状態（graph-states.json に追加）

- ローカル（自社線）とリモート（管理局）の**2段表示**を導入
- 約7〜8状態:
  - `ch2-local-only`: ローカルのみ（Ch1のmerged状態を引き継ぐ）
  - `ch2-remote-added`: リモートノード出現（空）
  - `ch2-pushed`: ローカル→リモートに矢印付きで同期
  - `ch2-pr-created`: PRノード（審査中マーク）
  - `ch2-review`: レビューコメント表示
  - `ch2-pr-merged`: PRがマージされた状態
  - `ch2-tagged`: リリースタグ付きノード
- **色**: リモート = 新色（例: `#a855f7` 紫系）、PR = `#ff9800` オレンジ

### 1-3. 凡例（legend）の更新

- `main 本線`（cyan）+ `origin/main 管理局`（紫）の2色

## 技術的な注意事項

1. **データ駆動**: 新章の追加は基本的に `chapters.json` と `graph-states.json` へのデータ追加のみ
2. **エンジン拡張が必要なケース**:
   - 第3章: コンフリクトマーカーのターミナル表示（ui.js）
   - 第4章: ノードラベル変更アニメーション（graph.js）
   - 第6章: 列車移動アニメーション（graph.js）
   - 第7章: スコア重み付けロジック（engine.js）
3. **グラフ設計**: 章が進むにつれグラフが複雑化する。SVGの viewBox 自動計算が正しく動くことを各章で確認
4. **モバイル対応**: 各章追加時にモバイルでの表示確認を行う（特にグラフの複雑化に伴う）
