# git train — 企画・設計ドキュメント

## 概要

「git train」は、鉄道の建設・運行をメタファーにしてGitを学ぶインタラクティブな学習ゲーム。路線図がリポジトリの状態を視覚的に表現し、ストーリーを進めながらGitコマンドを習得する。

### コンセプト

- **ローカル = 自社線**: 自分の管轄内で自由に列車（コミット）を走らせ、駅（ブランチ）を新設・廃止できる。誰にも影響しないから気軽に試行錯誤できる。
- **リモート = 相互直通で他社の運行が絡む世界**: 直通運転が始まった瞬間、自社だけの都合では動けなくなる。他社側でもダイヤが動いているから、自社の変更を送る（push）前に他社の最新状況を確認（fetch）して、必要なら取り込んで調整（pull）しないといけない。

### 鉄道メタファーの対応表

| Git概念 | 鉄道メタファー |
|---------|---------------|
| `git init` | 建設用地の確保（会社設立） |
| `git add` | 設計図を建設予定リストに登録（用地買収） |
| `git commit` | 駅の開業 |
| `.gitignore` | 立入禁止区域（工事現場の安全管理） |
| ブランチ作成 | 工事用支線を引く |
| `git merge` | 工事成果を本線に統合 |
| `git remote add` | 新たな直通運転協定を結ぶ |
| `git push` | 自社のダイヤを直通先に送る / 鉄道管理局への申請書提出 |
| `git pull` | 直通先のダイヤ変更を自社に取り込む |
| `git fetch` | 直通先の最新ダイヤを確認するだけ（自社に反映しない） |
| `force push` | 直通先のダイヤを自社の都合で強制上書き（大事故案件） |
| `git pull --rebase` | 直通先のダイヤに合わせて自社の発車順序を組み直す |
| コンフリクト | 同じ線路に同じ時刻で別の列車が入ろうとしている / 駅名論争（地元の反対） |
| `fork` | 他社の路線図をベースに自社路線を計画する |
| プルリクエスト | 「この列車を直通先にも走らせたい」という運行申請。相手がレビューして承認する |
| `git revert` | 該当区間の運行を止める |
| `git cherry-pick` | 必要な列車だけ別路線に振り替える（振替輸送） |
| `git reflog` | 30分前のダイヤに戻す（消してしまった運行記録を復元） |
| `git bisect` | どの時点から異常が起きていたかを特定する（障害の原因調査） |
| `git commit --amend` | 直前の駅名変更（仲木戸→京急東神奈川） |
| `git rebase -i` | もっと前の駅名を変更する |
| `git submodule` | 他社路線の買収（独自のダイヤを持ったまま自社の一部になる） |
| CI/CD | 自動信号システム / 列車が発車する前に自動検査する仕組み |
| ブランチ保護ルール | 安全運行規則 |
| issue | 住民からの問題提起・議論 |

## 全7章構成

### 第1章「git鉄道をつくろう」

駅を作って線路を引く。自社路線の建設が始まる。

- **学習内容**: `git init`, `.gitignore`, `git add`, `git commit`, `git switch -c`（ブランチ作成）, `git merge`
- **ストーリー**: まっさらな土地に自社路線を敷設する。init が会社設立、add が用地買収、commit が駅の開業。作業用ブランチは「工事用の別路線」。駅の線路、駅の建物など、工事に見立てる。うまくいったら main にマージして本線に採用。
- **`.gitignore`の伏線**: 工事現場の「立入禁止区域」として自然に登場。第7章で「あの時設定した安全策が実は大事だった」と伏線回収する。
- **グラフ**: main線のみ → 工事用支線が分岐 → マージで統合

### 第2章「鉄道管理局の承認を得よう」

国土交通省的な存在（ゲーム上では「鉄道管理局」）にGitHub経由で申請する。ここではまだスムーズに進む。

- **学習内容**: `git remote add`, `git push`, プルリクエスト, レビュー, マージ承認, リリースタグ（開通）
- **ストーリー**: 自社だけでは鉄道は開業できない。鉄道管理局（GitHub）の認可が必要。push が申請書の提出、プルリクエストが審査プロセス、レビューが当局のチェック、マージ承認で晴れて開通。リリースタグを打って開業日を記録。「自社の自由」から「外部との調整」へ意識が切り替わる。
- **ポイント**: この段階ではスムーズに進む設計。issueの必要性はまだ実感できない。
- **グラフ**: ローカル（自社線）とリモート（管理局）の2段表示が登場

### 第3章「新しい駅を作ろう！」

線路をそのまま延伸し、新しい駅を作る。駅名で大騒ぎが起きる。

- **学習内容**: issue管理, ラベル, コンフリクト発生, `git diff`, コンフリクト解消, 再push
- **ストーリー**: 延伸計画を立てる → 駅名案を出す → issueで反対意見が出る（「高輪ゲートウェイなんて名前は反対だ！」）→ 議論がラベルやコメントで整理される → 修正ブランチを切る → コンフリクト発生（地元住民＝他の開発者の変更と衝突）→ diff で対立を確認 → 解消して再push → issueクローズ。
- **issueの導入理由**: 第2章では順調だったからissueを立てる動機がない。第3章で初めて「意見が割れる」「問題が起きる」状況になり、「記録を残す仕組みが必要」とissueが自然に登場する。
- **グラフ**: 延伸で新しいノードが追加、コンフリクト状態の可視化

### 第4章「駅の名前を変えよう！」

開業済みの駅の名前を変える、あるいは途中駅を増やす/減らす。歴史改変が必要になる。

- **学習内容**: `git commit --amend`, `git rebase -i`, `force push`, 歴史改変の危険性
- **ストーリー**: 仲木戸→京急東神奈川のような駅名変更。amend は直前のコミットメッセージ修正。もっと前の駅名変更は rebase -i。「でも既に時刻表（リモート）に載ってたら？」→ 歴史改変後の force push の危険性。「開業済みの駅の名前を変えるのは大変」＝「push した後の rebase は危険」。幕張豊砂のような途中駅の追加/削除も扱う。
- **グラフ**: ノードのラベルが書き換わるアニメーション、force push の警告表示

### 第5章「他社の路線を買収しよう」

サブモジュールなどの高度な機能。

- **学習内容**: `git submodule`, `git subtree`, `fork`, `upstream`
- **ストーリー**: 買収した路線は自社の一部だけど、独自のダイヤ（コミット履歴）を持っている。更新タイミングも別管理。fork は「他社路線の設計図をもらって自社版を作る」。upstream の設定は「買収元の本家路線の動向もウォッチする」。
- **グラフ**: 自社線に別色の路線が接続される表現

### 第6章「自動運行システムを導入しよう」

CI/CDの導入。ここで初めて「電車が走り出す」。

- **学習内容**: CI/CD, GitHub Actions, 自動テスト, 自動デプロイ
- **ストーリー**: 路線が拡大して人手での運行管理が限界に。自動信号システム（CI）を導入して、列車が駅に入る前に安全チェックを自動実行。GitHub Actionsが「列車が発車する前に自動検査する仕組み」。テストが通らないと発車できない。CDは「検査通過後に自動で次の駅まで走らせる」＝自動デプロイ。これで初めてインフラの上を電車が自動で動き始める。
- **第1〜5章との転換点**: 第1〜5章は全部「静的な構造物」（インフラ整備）。第6章で初めて「その上を走る電車」（動く本番サービス）が出てくる。「静→動」の転換。
- **グラフ**: 列車アイコンが路線上を動く表現

### 第7章「緊急事態発生！ そして安全運行へ」

事故発生の対処と、事故を防ぐ方法を最後に学ぶ。

- **学習内容（前半・事故対応）**: `git revert`, `git cherry-pick`, `git reflog`, `git bisect`
- **学習内容（後半・防災体制）**: ブランチ保護ルール, 必須レビュー, CI必須化, `.gitignore`の伏線回収
- **ストーリー前半**: 沿線の変電所が火災で送電停止、一部区間が運行不能に。revert で問題のコミットを打ち消す（該当区間の運行を止める）。cherry-pick で必要な列車だけ別路線に振り替える（振替輸送）。reflog で「30分前のダイヤに戻す」（消してしまった運行記録を復元）。bisect で「どの時点から異常が起きていたか」を特定（障害の原因調査）。
- **ストーリー後半**: 「二度と起こさないために」— ブランチ保護ルール、プルリクエスト必須化、CIの必須チェック、`.gitignore`の伏線回収（「あの時の立入禁止区域設定が大事だった」）。最後に「安全運行宣言」を出して全路線の安全運行体制が確立、ゲームクリア。
- **最終章を「防ぐ方法」で締める理由**: 事故対処で終わると「Gitは怖い」で終わる。防止策で締めると「これで安心して運行できる」で前向きに終われる。
- **ゲーム性**: 第5章までは4択やコマンド入力で正解が一つ。第7章は複数の解法があって、選んだ手順によって復旧時間（スコア）が変わる形にすると難易度も体験も一段上がる。
- **グラフ**: 障害区間の赤い表示 → 復旧アニメーション → 安全運行体制の完成

## 全体の流れ

建設 → 認可 → 延伸 → 改称 → 買収 → 自動化 → 事故と防災

鉄道会社の成長物語として、Git学習のカリキュラムとして、綺麗に完結する。

章が進むごとに「自社内完結 → 外部との連携 → トラブル対応 → 歴史操作 → 高度な統合 → 自動化 → 非常時対応」と難易度もスコープも広がる。

### 難易度の段階

1. 第1〜2章: 正しい手順を覚える（基礎）
2. 第3〜4章: トラブル対応と判断（中級）
3. 第5〜6章: 高度な機能と自動化（上級）
4. 第7章: 正解がない状況で判断する（実践）

### キーとなる対比

- **第1〜5章**: 駅と線路のこと（静的な構造物＝リポジトリの「形」）
- **第6章**: 自動運行システム導入（静→動の転換点）
- **第7章**: 動く電車がある状態での障害対応（走りながら直す）

## 技術設計

### ファイル構成

```
git-train/
  index.html          シェル: レイアウト、CSS
  js/
    engine.js          ゲームエンジン: シーン遷移、スコア、セーブ/ロード
    graph.js           路線図描画: SVG生成、アニメーション
    ui.js              DOM操作: テキスト表示、選択肢、フィードバック
  data/
    chapters.json      全章のストーリーデータ
    graph-states.json  路線図の状態定義
```

### 技術選定

- **バニラJS**: フレームワーク不要。この規模ではかえって軽量。
- **データとロジックの分離**: chapters.json にストーリー、graph-states.json に路線図状態。章の追加は JSON 追加のみ。
- **3モジュール構成**: engine.js（遷移/スコア/セーブ）、graph.js（SVG描画）、ui.js（DOM操作）。

### レスポンシブ対応

- **PC**: 左右2カラム。左に路線図を大きく表示（flex: 3）、右にストーリーパネル。画面いっぱいに路線が増えていく体験を重視。
- **モバイル**: 縦1カラム。上から「路線図 → ストーリー/ターミナル → 選択肢」の順で積む。路線図は折りたたみ可能（タップで展開）。選択肢は画面下部に固定配置で片手操作しやすく。
- **切り替え**: CSS `@media (min-width: 768px)` で分岐。

### セーブ機能

- `localStorage` で進行状況を自動保存。
- 保存データ: 章インデックス、シーンインデックス、正解数、総問題数。
- ページを開いた時に保存データがあれば「前回の続きから乗車しますか？」と表示。

### 路線図（SVG）

- グラフ状態は JSON で定義。各状態にノード（駅）、ライン（線路）、ラベル（ブランチ名）を持つ。
- ノード追加時のアニメーション（フェードイン）。
- glow エフェクトで最新のノードを強調。
- 破線ノードでステージング状態（未確定）を表現。

### ストーリーデータ構造（chapters.json）

各章は以下の構造:

```json
{
  "id": 0,
  "title": "第1章",
  "subtitle": "git鉄道をつくろう",
  "description": "章の説明",
  "legend": [{ "color": "#00c8ff", "label": "main 本線" }],
  "scenes": [
    {
      "char": "キャラクター名",
      "text": "表示テキスト（HTMLタグ可）",
      "terminal": "ターミナル出力（nullで非表示）",
      "choices": [
        { "label": "選択肢テキスト", "correct": true/false }
      ],
      "feedback": { "ok": "正解時メッセージ", "ng": "不正解時メッセージ" },
      "graph": "グラフ状態キー",
      "nextLabel": "次へボタンのテキスト（省略時は「次の駅へ」）",
      "isLast": true  // 章末のシーンに付与
    }
  ]
}
```
